## 1. Agent participation — "did any agent watch when we entered?"

|---|---|

The 14 missing items (29 → 15) are utility modules (DeterministicFallback, FreeOnChainDataProvider, AgentBase itself, etc.) — **not** stand-alone agents.

In the backtest record, each trade's `entryAgentVotes` array typically contains 2–4 voting agents — that's the honest count, not 29.

## 2. Cross-symbol — "are we using BTC vs ETH vs SOL relation?"

**No.** Cross-symbol intelligence is built but not wired into trade decisions.

- `Week9RiskManager.ts:451-574` enforces *static* CORRELATION_GROUPS (BTC/ETH treated as 0.70 correlated by hard-coded rule). Used only to **block** correlated positions when exposure > 30%. Doesn't predict.
- `DynamicCorrelationTracker.ts` is a real rolling-Pearson tracker (5-min returns, 24-h window). **`GlobalSymbolAnalyzer` never instantiates it. Dead code.**
- Per-symbol agents are independent. The BTC-USD TechnicalAnalyst has zero knowledge of ETH-USD/SOL-USD signal state.
- `ForexCorrelationAgent` only handles forex pairs (EURUSD/GBPUSD) — not crypto cross-pairs.

This is why dropping BTC (scenario AE → +20.11%) worked: BTC was just consuming capital, not contributing leading-indicator value.

## 4. Latency — "did entry/exit happen at exact point?"

End-to-end tick → trade:

| Phase | Paper | Live |
|---|---|---|
| WS tick ingest | <5 ms | <5 ms |
| Agent fast-path compute | 50–200 ms | 50–200 ms |
| Consensus + R:R + ML gate | 25–60 ms | 25–60 ms |
| Order placement | 50–200 ms (sim) | 100–500 ms (REST) |
| **Total** | **~150–500 ms** | **~300–1000 ms** |

`PositionManager.ts:137-143` polls exits every **100 ms** (fixed `setInterval`) — not per-tick. So exits can be up to 100 ms late after a level is breached.

`GlobalSymbolAnalyzer.ts:613` sets a **120-second timeout** on slow agents (separate timer, not hot path).

`PaperTradingEngine.ts:508-512` has a `PRICE_ZERO_BLOCKED` guard, and `PositionManager.ts:193-198` warns when price > 10 s old → direct evidence the system has had stale-price incidents.

**Backtest blind spot:** `yearly-backtest.ts` resolves at 15-min candle close = 900 000 ms decision interval. Live PriorityExitManager runs at 100 ms. The 9000× granularity gap means backtest cannot see intra-candle whipsaws (most stop-outs).

## 5. Loss anatomy on N (1182 losses out of 1881)

| Exit reason | Count | % of losers |
|---|---|---|
| STOP_LOSS_HIT | 650 | **55%** |
| THESIS_INVALIDATED | 238 | 20% |
| STUCK_POSITION | 176 | 15% |
| CATASTROPHIC | 118 | 10% |

Split a different way:

- **343 losers (30%)** had peak unrealized ≥ +0.5% then reversed (avg peak +1.04%, avg final -1.19%) → **exit-timing failure**
- **359 losers (30%)** never showed any profit (peak = 0) → **entry-quality failure**

**Killer finding:** average consensus.strength on winners ≈ 0.988; on losers ≈ 0.988. **Indistinguishable.** That's why the Phase 43 ML test got **test AUC 0.463** (below chance OOS) — the entry features genuinely don't separate winners from losers.

**Time-of-day clusters with 70-73% loss rate:** UTC hours 02, 10, 20, 23 — Asia open, London open, US afternoon, US close. The Phase 43 time-filter only caught 18-21 UTC, missed the others.

**BTC-specific:** 60% of BTC losses are STOP_LOSS_HIT (vs portfolio average 55%) — fixed ATR-based stops bite harder on BTC's lower vol.

## 6. Small profits — "why are wins tiny?"

- Avg win **+1.80%** vs avg loss **-0.86%** = payoff 2.1×
- Avg intraday range on BTC/ETH/SOL is 2-3% → we harvest ~60% of the available move
- Long-hold winners (>4 h): **+2.04%**
- Short-hold winners (≤30 min): **+1.29%** — these are leaving money on the table

Phase 22 walked-TP fires at lazy near-S/R levels. The walker should reject targets < 1×ATR from entry.

## 7. ms-prediction — "why can't we predict in ms?"

We can't physically — 14+ agents × 50-300 ms compute + slow API calls + 100-500 ms exchange round-trip = realistic floor 150 ms paper / 300 ms live.

**But latency isn't the bottleneck.** Even at 150 ms tick-to-trade, the consensus is 50/50 (AUC 0.46). Faster execution of a non-predictive signal yields noise faster, not money.

The real gap is **information**, not **speed**:
- OrderbookImbalance is a 5-bar volume stub, not real L2
- Whale Alert stubbed without key
- Dune Analytics disabled
- News is RSS/cache, not real-time wire
- No social-media stream
- No cross-symbol leading indicators

## 8. The action list

Ordered by leverage. None of these are parameter tuning — they are plumbing real information into the agents.

1. **Wire `DynamicCorrelationTracker` into the consensus path** — dead code today; just needs `globalSymbolAnalyzer.attachCorrelationTracker(tracker)` and a consensus-bias hook. ~1 day.
2. **Set `WHALE_ALERT_API_KEY` + `DUNE_API_KEY` + `ENABLE_ONCHAIN_FLOW_ANALYST=true`** — zero code change. Whale-flow at 4-8h horizon is the documented edge funding-rate isn't providing.
3. **Real L2 orderbook into OrderbookImbalance** — replace the 5-bar volume stub with the actual Coinbase/Binance L2 stream. Highest single new-information source.
4. **Time-of-day expansion** to skip 02/10/20/23 UTC clusters too — 30 min.
5. **Walked-TP minimum-reach** ≥ 1×ATR — 2 hours.
6. **Per-symbol ATR-stop multipliers** — `ARG_SL_ATR_BTC=2.5 ARG_SL_ATR_SOL=1.8` style. 30 min.
7. **Real-time news wire feed + LLM classifier** — multi-week build, but largest unclaimed alpha pool.

The first 3 are highest leverage. Until those land, the **37% WR backtest ceiling is real**, the ML test confirmed why (no separating signal in entry features), and **all the parameter sweeping in the world won't lift it**.
